Technical Terms — Simple Explanations
FastAPI + WebSocket API
REST API A standard way for the app and server to talk to each other over the internet — like a waiter taking your order and bringing back food
WebSocket A live open connection between browser and server — server can push updates instantly without the browser asking repeatedly
HTTP Polling The browser asks the server "is it done yet?" every few seconds until it gets a final answer — simpler alternative to WebSocket
RBAC Role-Based Access Control — each user can only do what their role allows. Estimator uploads, Manager comments, Admin configures
Uvicorn The engine that runs the FastAPI server — handles many requests at the same time efficiently
Azure Service Bus
Async job queue A waiting line for jobs — tasks are placed in the queue and picked up by workers one by one, so the user doesn't have to wait
Dead-letter queue A separate holding area for jobs that failed too many times — kept for investigation instead of being silently lost
max_delivery_count Maximum number of times a failed job is retried before moving to the dead-letter queue — set to 3
lock_timeout How long a worker has to finish a job before it's released back to the queue for another worker to pick up — set to 5 minutes
KEDA Workers · Celery
KEDA Kubernetes Event-Driven Autoscaling — automatically starts more workers when the queue is busy and shuts them down when idle
group() fan-out Splits one job into many smaller tasks running at the same time — all 15 pages of a drawing are processed simultaneously, not one by one
asyncio.gather() Fires multiple async calls at exactly the same time — all pages of a drawing are processed simultaneously instead of waiting for each other
chord() fan-in Waits for all parallel page tasks to finish, then triggers the next step (Reconciler) — like waiting for everyone to finish before moving on
Reconciler Merges the outputs from Zeus into one final result — flags low-confidence extractions as items needing human review
Deterministic pricing Pricing calculated by fixed rules and rate cards — no AI involved, same input always gives exactly the same price
PostgreSQL 16
HA + PITR High Availability + Point-In-Time Recovery — system stays up even if a server fails, and can be restored to any exact moment in history if needed
SQLAlchemy 2.0 async The tool that lets the application talk to the database without waiting — other tasks continue while the database responds
Redis Cache
Cache-hit When the same drawing is submitted again, the result is found in cache and returned instantly — no AI models are called again
Zero model cost If a result is served from cache, no AI API is called — meaning no token charges for that request
Knowledge Graph DB
SIMILAR_TO edges Links between items across different bids that look alike — helps the system learn that similar drawings usually have similar results
Few-shot grounding Showing the AI a few real past examples before asking it to extract — like giving someone 3 solved examples before an exam
prompt_version A unique ID stamped on every AI instruction set — so every result can be traced back to the exact instructions the AI was given
Golden set A collection of verified correct examples used to test AI accuracy — like an answer key used to grade the AI before it goes live
General
DPA Data Processing Agreement — a legal contract ensuring the AI provider handles data safely and doesn't use it for training their models
Constrained JSON schema A strict template the AI must fill in — ensures output is always structured and predictable, never free-form text
